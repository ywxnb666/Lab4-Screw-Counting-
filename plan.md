# 视频螺丝计数方案设计（修订版）

## 1. 任务目标与约束

对输入文件夹中的每段视频完成：

- 统计 5 类螺丝总数，输出到 `result.npy`
- 导出 1 张展示用掩膜叠加图 `{video_name}_mask.png`
- 记录总耗时到 `time.txt`

程序入口统一为：

```bash
python run.py --data_dir /path/to/test_videos_folder --output_path ./result.npy --output_time_path ./time.txt --mask_output_path ./mask_folder/
```

任务核心难点：

1. 同一颗螺丝在多帧中反复出现，必须避免重复计数
2. 螺丝是小目标，存在接触、密集、遮挡
3. 5 类分类需要准确，Lab2 分类器迁移到视频 crop 后性能会下降
4. 需要兼顾速度（10s 视频处理 ≤10s）

设计原则：

- **简单优先**：能用一条路线解决的不分两条，能用成熟工具链的不自己造
- **尽早跑通**：第 1 周必须有能出数字的端到端 baseline
- **分类是关键**：检测和去重再好，分类错了就白做

## 2. 方案总览

### 2.1 核心架构

```
视频读取 → 方向修正 → 关键帧提取
                          ↓
                   锚帧配准（homography）
                          ↓
                   one-class 螺丝检测（YOLO）
                          ↓
                   全局坐标去重聚类
                          ↓
                   5 类分类投票
                          ↓
                   输出 result.npy / mask / time.txt
```

只有一条主路线：**几何去重计数**。

样例视频全部是静态桌面扫拍，作业说明盲测"场景特征依然符合预期规范"，不太可能出现传送带等动态场景。不再为此设计完整的 Route B。

### 2.2 Tracker 的定位：局部时序关联辅助，不独立计数

Tracker **不是一条独立的计数路线**，而是主路线的辅助工具。

原因：静态场景下，同一颗螺丝随相机扫动离开画面后再次进入，ByteTrack 会分配新 track ID，导致"每条 track 计一次"严重 over-counting。因此 tracker 不能独立负责最终计数。

Tracker 的正确用法：

- **短时关联**：在相邻帧之间维持检测 ID 连续性，补偿偶尔的检测丢失
- **辅助选 crop**：同一 track 的多帧观测中挑最佳 crop 给分类器
- **配准局部失败时的降级策略**：如果某段关键帧配准失败（inlier 率极低），用 tracker 维持这几帧内的局部关联，但这些 track 仍然需要尝试投影到参考坐标做历史去重——只是投影精度降低，不是放弃几何去重

实现上直接用 YOLO 内置 ByteTrack（`model.track()` 一行调用），不需要额外开发。但最终计数始终以"参考坐标系中的唯一 cluster 数"为准，tracker 只是帮助建立观测之间的对应关系。

## 3. 核心模块设计

### 3.1 视频预处理与关键帧选择

**视频读取：**

- 用 OpenCV VideoCapture 读取
- 检查并处理旋转元数据（手机视频的 rotation flag）
- 保留两套分辨率：
  - 低分辨率（长边 960）：用于配准
  - 原始分辨率：用于检测和分类 crop

**关键帧选择：**

不用复杂策略，采用简单有效的规则：

- 计算相邻帧的光流位移均值（或 ORB 特征点位移均值）
- 当累积位移超过阈值（如画面宽度的 15-20%）时，选为关键帧
- 保底：最少每 15 帧取 1 帧，最多每 5 帧取 1 帧
- 典型 10s/30fps 视频预计提取 20-40 个关键帧

### 3.2 锚帧配准（替代逐帧累积）

**逐帧链式累积 homography 的核心问题是误差累积**。改用锚帧直配策略：

#### 锚帧选择

- 从关键帧中均匀选取 4-6 个锚帧（如第 1、25%、50%、75%、100% 位置）
- 要求相邻锚帧之间有足够重叠（≥30% 画面重叠）

#### 配准策略

- 每个关键帧直接配准到最近的锚帧（不链式传递）
- 所有锚帧之间再做一次配准，建立锚帧间的变换关系
- 最终所有关键帧都可以通过"关键帧→最近锚帧→参考锚帧"投影到统一坐标系
- 这样最多只有 2 次矩阵乘法，误差累积大幅减少

#### 特征匹配实现

- 在低分辨率灰度图上提取 AKAZE 特征（比 ORB 更稳定）
- knnMatch + ratio test（ratio=0.75）
- RANSAC + findHomography（阈值 5.0）
- 记录 inlier 比例，低于 0.3 标记为配准失败

#### 配准质量监控

对每对配准结果记录：

- inlier 比例（< 0.3 标记失败）
- 重投影误差均值
- 匹配特征点数量

当某个关键帧配准失败时：

1. 尝试对次近锚帧配准
2. 若仍失败，该帧检测结果不参与聚类，但通过 tracker 短时关联保留信息
3. 在日志中记录失败位置，供调试

### 3.3 one-class 螺丝检测

检测是整个系统的共享底座。

**模型选择：**

- YOLOv8s 或 YOLOv11s（speed/accuracy 平衡点好）
- 统一只检测一个类别：`screw`
- 1080p 视频直接整图推理
- 4K 视频启用 SAHI 切片推理（切片 640×640，overlap 20%）

**训练数据：**

- 从 3 个开发视频中抽 80-120 张关键帧
- 所有螺丝统一标 bbox 为 `screw`
- 半自动标注流程：
  1. 先手工标 30 张核心样本
  2. 训练初版 detector
  3. 用初版模型预标剩余帧
  4. 人工修正漏检和错检
- 4 人分摊标注，1 人统一质检

**推理优化：**

- 使用 FP16 推理
- batch_size=1 逐帧推理（显存友好）
- NMS IoU 阈值 0.5，confidence 阈值 0.25（宁可多检不漏检，后面靠聚类去重）

### 3.4 全局去重聚类

对每个关键帧的检测结果：

1. 提取 box center、box size、confidence
2. 通过 homography 投影到参考坐标系
3. 在参考坐标系中做聚类

**聚类方法：**

- 增量贪心匹配（不用复杂聚类算法）：
  - 对新检测到的每个 box，在已有 cluster 中寻找最近邻
  - 匹配条件：投影后中心距离 < 阈值（如 box 对角线的 0.5 倍）且面积比在 [0.5, 2.0] 范围内
  - 匹配上则更新 cluster 记录
  - 未匹配上则新建 cluster

- 每个 cluster 维护：
  - 所有观测记录（帧号、原图坐标、投影坐标、confidence）
  - 最佳 crop（选 confidence 最高、清晰度最好的一次观测）
  - 分类得分累积向量

- 最终 cluster 过滤：
  - 只保留被观测到 ≥2 次的 cluster（过滤 false positive）
  - 每个保留的 cluster 计数 1 次

### 3.5 5 类分类（关键路径——决定最终得分上限）

**检测和去重决定下限，分类决定上限。** 即使总数完全数对，分类错一个就丢 2-4 分。10 个计数值中只要有 3 个分类错导致误差 ≥4，就直接丢 12 分。因此分类模块是整个系统的关键路径，必须最早验证、最多投入。

分类不是逐帧逐框独立做，而是 cluster 级别做：

#### 分类器准备

1. **第 1 周就必须验证 Lab2 模型在视频 crop 上的 baseline 准确率**。如果直接用效果 <80%（预期如此，因为 Lab2 数据太干净），则域适应 fine-tune 不是可选项，是必做项
2. **域适应 fine-tune 流程**：
   - 从样例视频中切出检测 crop，按类别手动分拣（每类 30-50 张）
   - 与 Lab2 原始数据混合（比例约 1:1）
   - 增强策略：随机高斯模糊、亮度/对比度抖动、随机旋转 0-360°、木桌背景混合
   - 如果某类样本不够，优先用该类数据做更多增强而不是降低采样
3. 模型结构：轻量 CNN（如 ResNet18 / EfficientNet-B0 / MobileNetV3）
4. 输入尺寸：统一 resize 到 64×64 或 96×96

#### Cluster 级投票

单帧分类容易受模糊、遮挡、角度影响。cluster 级投票通过多帧聚合来提升鲁棒性：

- 对同一 cluster 收集 top-k 个最佳 crop（按 confidence × 清晰度排序，k=3~5）
- 每个 crop 过分类器得到 5 维概率向量
- 加权求和（权重 = detection confidence × crop 清晰度得分）
- 取 argmax 作为最终类别
- 记录投票置信度（最高概率与次高概率之差），低置信 cluster 标记为需人工检查

#### 混淆分析（第 2 周结束前必须完成）

- 在 3 段开发视频上跑完后，统计 5×5 类别混淆矩阵
- 重点关注：
  - 哪两类最容易混淆 → 针对性增加训练样本
  - 哪类 recall 最低 → 检查是否是 crop 质量问题还是类间相似度问题
  - 投票置信度分布 → 低置信 cluster 是否集中在某类
- 混淆矩阵直接作为报告素材

### 3.6 Mask 输出

mask 只用于展示，不影响计数。按优先级递减选择：

**方案 A（推荐）：SAM2 单帧增强**

- 选择 cluster 观测最密集的一帧作为展示帧
- 以检测 box 作为 SAM2 的 prompt
- 生成像素级 mask 叠加到原图
- 只对 1 帧做 1 次，速度影响可忽略

**方案 B（保底）：检测框 + 类别标注**

- 在展示帧上画出检测框
- 标注类别和 confidence
- 最简实现，10 分钟搞定

开发阶段先用方案 B 确保有输出，收尾阶段换方案 A 提升展示质量。

## 4. 全景拼接实验分支（第 1 周验证，不作为默认主线）

全景拼接理论上能天然去重，但存在未验证的风险，**不能把项目成败押在它上面**。主线始终是第 3 节的锚帧配准 + 聚类去重。

### 4.1 为什么不做主线

- OpenCV Stitcher 偏向"视觉上拼得好看"，不保证小目标的几何位置足够精确
- 螺丝是小金属件，拼接边缘的变形、鬼影、局部错位会直接影响检测和分类
- 4K 视频拼接后全景图可能极大，检测推理和显存都有压力
- 还没有在这些视频上验证过拼接成功率

### 4.2 第 1 周验证流程

```
关键帧提取 → OpenCV Stitcher 拼接 → 全景图上检测 → 与主线结果对比
```

验证标准：

- 3 段开发视频是否都能拼接成功（不报错、不崩溃）
- 拼接后检测出的螺丝总数是否与主线接近
- 小目标是否在拼接边缘处出现明显变形或丢失

### 4.3 结论处理

- 若 3/3 成功且检测结果优于主线 → 升级为可选加分路线，在消融实验中作为一个对比方案
- 若拼接不稳定或检测效果差 → 仅在报告中作为 negative result 记录，不进入最终提交代码

## 5. 速度策略

### 5.1 时间预算分配（10s 视频目标 ≤10s）

| 模块 | 预算 | 策略 |
|------|------|------|
| 视频解码 + 方向修正 | 0.5s | OpenCV 直读 |
| 关键帧选择 | 0.5s | 轻量光流/特征点 |
| 配准/拼接 | 1-2s | 低分辨率流 |
| 检测（YOLO） | 3-5s | 关键帧 only，FP16 |
| 分类 | 0.5-1s | cluster 级，轻量 CNN |
| Mask 生成 | 1-2s | 单帧 SAM2 或 bbox |
| 输出写入 | 0.1s | - |
| **合计** | **~7-11s** | |

### 5.2 硬件适配

**4060 8G 开发环境：**

- YOLO s 档模型 + FP16 足够
- 关键帧策略控制推理次数在 20-40 次
- 分类用 batch 推理加速

**若有 H200：**

- 用于 detector 和 classifier 训练加速
- SAHI 参数搜索
- SAM2 mask 生成

### 5.3 第 1 周必须完成速度 benchmark

在 4060 8G 上对一段 4K 样例视频跑一遍完整链路，记录各模块耗时。即使 detector 还是初版，也要摸清速度瓶颈。

## 6. 数据策略

### 6.1 检测标注

| 阶段 | 内容 | 负责 |
|------|------|------|
| 第 1 步 | 从 3 段视频抽 80-120 张关键帧 | D |
| 第 2 步 | 4 人各标 20-30 张 bbox（统一为 `screw`） | 全员 |
| 第 3 步 | B 统一质检，修正标注 | B |
| 第 4 步 | 训练初版 detector，预标剩余帧 | B |
| 第 5 步 | 全员修正预标结果 | 全员 |

工具推荐：Roboflow 或 CVAT（在线协同标注）。

### 6.2 分类数据

**关键：不能只靠 Lab2 数据。**

- 从视频检测结果中切出 crop，按类别手动分拣
- 每类收集 30-50 张视频 crop + Lab2 原始数据混合
- 做域适应增强：
  - 随机高斯模糊（模拟运动模糊和对焦不准）
  - 亮度/对比度随机抖动
  - 随机旋转 0-360°
  - 随机缩放 0.8-1.2×

### 6.3 不建议做 5 类检测

5 类检测标注成本是 one-class 的 5 倍，且分类投票已经能提供足够的类别信息。除非分类投票在开发视频上表现极差（>20% 类别错误率），否则不做。

## 8. 项目推进顺序

### 第 1 周：端到端 baseline 跑通 + 分类 baseline 验证

**目标：能出数字，能提交，哪怕精度差。同时拿到分类器的 baseline 数据。**

- A：视频读取 + 方向修正 + 关键帧提取 + 锚帧配准原型
- B：制定标注规范，组织首批标注（30 张手标）并训练初版 one-class detector
- C：整理 Lab2 分类器 + 从视频手切 50-100 张 crop 做 baseline 测试 + 记录 5 类准确率 → **如果 baseline <80% 立即启动 fine-tune**
- D：搭建抽帧/导出工具 + `run.py` 骨架 + 输出格式封装 + 第一次 4K 速度 benchmark + 全景拼接实验

**第 1 周结束交付物：**
- 一个能跑通 `run.py` 的版本（精度不要求）
- 速度 benchmark 报告
- Lab2 分类器在视频 crop 上的 baseline 准确率报告
- 全景拼接可行性结论（实验分支）

### 第 2 周：检测和去重做稳

- A：完善锚帧配准 + 聚类去重实现（主线）
- B：扩充标注 + 训练正式 detector + 4K SAHI 调参
- C：从视频切 crop 做分类器 fine-tune + cluster 级投票实现
- D：集成检测 + 去重 + 分类全链路，开始记录误差，并整理全景拼接实验对比结果

**第 2 周结束交付物：**
- 3 段开发视频的计数结果 + 误差分析
- 分类器混淆矩阵

### 第 3 周：精度调优 + 展示

- A：修正配准失败样例，调整聚类阈值
- B：补充标注，提升 detector recall
- C：针对混淆类别增加训练样本，调投票权重
- D：实现 SAM2 mask 输出 + 消融实验记录 + benchmark 汇总

**第 3 周结束交付物：**
- 3 段开发视频精度接近最终水平
- 消融实验数据
- mask 展示图

### 第 4 周：收尾提交

- 全员联合测试 `run.py` 完整流程
- 确认输出格式严格符合规范（`.npy` 字典、mask 命名、time.txt）
- tracker 兜底路线快速验证
- 撰写个人报告 + 准备答辩

## 9. 四人分工

### 9.1 角色划分

分工原则：

- 每个人尽量只负责一条技术线
- 每条线都能独立形成明确交付物
- 通用机械工作由全员分摊，但技术决策只由对应 owner 负责

#### A：几何配准与去重

- 视频预处理、方向修正
- 关键帧选择
- 锚帧配准
- 全局去重聚类
- tracker 辅助关联
- 最终计数逻辑

职责边界：

- 只负责“跨帧对应和去重”
- 不负责 detector 训练
- 不负责分类器训练

#### B：检测数据与 detector

- 制定标注规范
- 组织标注、质检
- 训练 one-class YOLO detector
- SAHI 适配与调参

职责边界：

- 只负责“把螺丝找全”
- 不负责几何去重
- 不负责分类器训练

#### C：分类与结果融合

- Lab2 分类器迁移 + 域适应 fine-tune
- Crop 生成策略
- Cluster 级分类投票
- 混淆分析与改进

职责边界：

- 只负责“把唯一实例分对类”
- 不负责 detector 训练
- 不负责几何配准

#### D：工程工具、封装与评测

- 关键帧抽取、crop 导出、标注转换等数据工具
- `run.py` 统一入口
- 输出格式封装（npy / time.txt / mask）
- 速度 benchmark
- 消融实验记录
- 报告素材整理

职责边界：

- 只负责“让系统可跑、可测、可交”
- 不负责 detector / classifier 训练
- 不负责几何算法核心实现

### 9.2 协作接口

所有模块通过以下数据结构通信：

```python
# detector 输出
@dataclass
class Detection:
    frame_id: int
    bbox: np.ndarray      # [x1, y1, x2, y2]
    confidence: float
    crop: np.ndarray       # 裁切的原图区域

# 配准输出
@dataclass
class Registration:
    H_to_ref: np.ndarray   # 3x3 homography
    valid: bool
    inlier_ratio: float

# cluster 输出
@dataclass
class Cluster:
    cluster_id: int
    observations: list     # List[Detection]
    best_crop: np.ndarray
    class_probs: np.ndarray  # [5,] 累积分类概率
    pred_class: int

# 最终输出
counts: dict  # {"video_name": [c1, c2, c3, c4, c5]}
```

第 1 周先约定接口，各自独立开发，第 2 周开始集成。

标注分摊规则：

- 全员都参与基础 bbox 标注
- B 负责标注规范和最终质检
- D 负责抽帧、导出和标注格式转换工具

这样既能均摊重复劳动，也能保证标注口径统一。

## 11. 风险清单与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 分类器迁移效果差 | 高 | **极高** | 第 1 周验证 baseline，<80% 立即启动 fine-tune，这是最高优先级 |
| Homography 漂移导致重复计数 | 中 | 高 | 锚帧直配限制累积误差 ≤2 次乘法；配准失败帧用 tracker 短时关联兜住 |
| 4K 检测漏小目标 | 中 | 中 | SAHI 切片推理 |
| 盲测出现动态场景 | 低 | 高 | tracker 辅助 + 仍尝试几何去重，不做纯 track 计数 |
| 全景拼接失败 | 中 | **低**（已降为实验分支） | 不影响主线，仅影响消融实验中的一个对比方案 |
| 速度超限 | 低 | 低 | 关键帧策略 + FP16 已有足够余量 |
| 标注量不足 | 中 | 中 | 半自动标注流程 + 全员分摊 |
